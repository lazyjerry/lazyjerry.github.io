# Step 03 - 檢查與修正規格格式

> 提案建立後，打開 `openspec/changes/[name]/specs/` 下的 spec.md 檢查格式。
> 格式錯誤會導致驗證失敗、歸檔時合併出錯，務必在驗證前先確認。

```
請檢查這次提案的 specs 格式是否符合 OpenSpec 規範，有錯請修正
```

或是手動檢查：

### 檢查 1：每個 Requirement 是否有 Scenario

打開 spec.md，確認每個 `### Requirement:` 底下至少有一個 `#### Scenario:`。

```markdown
### Requirement: User Login
使用者 SHALL 能夠使用 Email 和密碼登入系統。

#### Scenario: 登入成功           <!-- 必須用 #### 開頭 -->
- WHEN 使用者輸入正確的 Email 和密碼
- THEN 系統回傳 JWT token
```

### 檢查 2：需求描述是否包含 SHALL 或 MUST

每條需求的描述句中必須出現 `SHALL` 或 `MUST`，這是 OpenSpec 驗證的必要條件。

### 檢查 3：Delta 標記是否正確

提案中的 spec.md 是差異格式（Delta），用以下標題區分變更類型：

| 標題 | 用途 | 注意事項 |
|------|------|----------|
| `## ADDED Requirements` | 新增需求 | 寫完整的新需求 |
| `## MODIFIED Requirements` | 修改既有需求 | **必須貼修改後的完整內容**，不能只寫差異 |
| `## REMOVED Requirements` | 移除需求 | 附上 Reason 和 Migration |
